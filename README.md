# Hard Rules, Soft Preferences: Personalized Packing Checklist Generation

Code and data release for the paper:

> **Hard Rules, Soft Preferences: Bridging Reasoning, Learning, and Optimization for Personalized Packing Checklist Generation**  
> Himel Dev, Madhusudan Basak, Tanmoy Sen, Paromita Shome, Bashima Islam  
> CIKM 2026 Applied Research Track

The system generates personalized, regulation-compliant packing checklists for air travel through three sequential stages:

1. **Symbolic Reasoning** — a forward-chaining rule engine (226 declarative rules) generates a high-recall seed checklist that is guaranteed compliant with TSA/FAA regulations and safety requirements.
2. **Preference Learning** — a gradient boosted inclusion model (GBM) and a LambdaMART ranker learn per-item relevance and priority from user edit logs, mitigating survivorship bias by separating the two signals.
3. **Constrained Optimization** — a CP-SAT solver selects a compact, feasible subset and assigns items to bags, maximizing expected retained utility under physical capacity, regulatory placement, and inter-item dependency constraints.

The system is deployed in **[FlyEnJoy](https://apps.apple.com/app/id6745104853)**, a production iOS travel app. Deployment doubled checklist completion rates and reduced per-checklist editing and completion time.

## Quick start

Run the full Stage 2 training pipeline on the 50-trip sample:

```bash
python train_pipeline.py
```

This will:
- Load the 50 sample trips from `data/samples/`
- Engineer features combining symbolic provenance, trip context, and item attributes
- Train the GBM inclusion model and LambdaMART ranker with 5-fold cross-validation
- Save models to `models/` and evaluation results to `results/`

Expected output on the sample (50 trips — numbers will differ from the paper's full 604-trip results):

```
Inclusion model: AUC-ROC ~0.85–0.90
Ranking model:   NDCG@5 ~0.88–0.92
```

To run the Stage 3 optimizer on a trip, use the `optimization_model.py` module directly:

```python
import sys; sys.path.insert(0, 'src')
from data_loader import load_data
from optimization_data import load_optimization_data
from optimization_model import PackingOptimizer

loader = load_data('data')
trip = loader.trips[0]
opt_data = load_optimization_data('data', trip)
optimizer = PackingOptimizer(opt_data)
solution = optimizer.solve()
print(solution.summary())
```

## Data

### Released data (10 sample trips)

`data/samples/user_context_sample.json` and `data/samples/user_evaluation_sample.json` contain 10 anonymized trip scenarios illustrating the range of inputs and outputs:

| Dimension | Distribution |
|---|---|
| Purpose | Leisure 3 · Adventure 3 · Business 2 · Family Visit 2 |
| Luggage | Carry-On 5 · Checked 4 · Personal 1 |
| Special needs | With Infant · Accessibility · Senior · Athlete (×2) · Pregnancy (×2) · Medical (×2) |
| Unique users | 3 (anonymized as user_001 … user_003) |

The full dataset (604 labeled trips, 20 labelers) is proprietary and not released. The sample is sufficient to run the training pipeline and verify all three stages end-to-end.

Real user IDs are replaced with anonymous identifiers. Timestamps are truncated to date only. Trip context fields (destination, activities, duration, special needs) are retained as-is; they are not personally identifiable.

Each evaluation record includes:
- **`seed_items`**: the checklist generated by the symbolic engine for this trip context
- **`user_selections`**: items the labeler retained, in priority order
- **`rule_activations`**: which rules fired (used as symbolic provenance features in Stage 2)

### Full dataset (not released)

The full dataset contains 604 labeled trips from 20 labelers. It is not released to protect labeler privacy. Researchers who wish to reproduce the full paper results may contact the authors.

`data/items/` contains an example item catalog (42 items across 7 categories) with physical attributes. The full production catalog (378 items) is proprietary.

`data/rules/` contains example rules (3 per family, 21 total) illustrating the rule format across all 7 families. The full rule base (226 rules) is proprietary.

## Stage 1: Symbolic Reasoning

The symbolic engine is implemented in the FlyEnJoy iOS app (Swift). The YAML rule catalogs in `data/rules/` are the complete, human-readable specification of the engine's knowledge base. Each rule has the form:

```yaml
- id: rule.tsa.liquids_311
  priority: 90
  when:
    - has_carry_on
  then:
    - recommend: [LIQ-ZIPLOCK-BAG]
    - fact: has_liquids_311_bag
  reason: "TSA 3-1-1 liquids rule applies for carry-on luggage"
```

The engine applies rules via priority-based forward chaining: high-priority regulatory and safety rules fire first, emitting item recommendations and derived facts that trigger subsequent rules. The seed checklists stored in `data/samples/user_evaluation_sample.json` are the outputs of this engine for each sample trip, enabling evaluation of Stages 2 and 3 without re-running the iOS engine.

**Stage 1 results on the full 604-trip dataset** (from the paper):
- Mean recall: 99.7%
- Mean precision: 75.7%
- Perfect recall on 96.0% of trips

## Stage 2: Preference Learning

Run `python train_pipeline.py` to reproduce Stage 2.

**Inclusion model** — GBM (gradient boosted trees, depth=5) predicts P(item retained | trip context, symbolic provenance). Features combine:
- Item attributes: category, weight, value tier, criticality
- Trip context: duration bin, purpose, travel type, luggage type, activities, temperature
- Symbolic provenance: which rule families activated the item, rule priority, rule count

**Ranking model** — LambdaMART learns relative priority among retained items using pairwise preference labels derived from the user's final ordering.

**Results on full dataset** (5-fold cross-validation):

| Model | Metric | Score |
|---|---|---|
| GBM (inclusion) | AUC-ROC | 0.943 |
| GBM (inclusion) | Recall@20 | 0.813 |
| LambdaMART (ranking) | NDCG@5 | 0.923 |
| LambdaMART (ranking) | NDCG@10 | 0.899 |

## Stage 3: Constrained Optimization

The CP-SAT optimizer selects items and assigns them to bags to maximize expected retained utility minus expected monetary loss, subject to:
- **Physical constraints**: weight and volume capacity per bag
- **Regulatory constraints**: e.g., lithium-ion batteries barred from checked baggage
- **Environmental constraints**: temperature-sensitive items must go in accessible bags
- **Dependency constraints**: must-together groups (camera + battery + lens) and must-separate groups (liquids + electronics)

**Results on full 604-trip dataset**:

| Method | Constraint satisfaction | Mean solve time |
|---|---|---|
| CP-SAT (ours) | **100%** | 340 ms |
| Greedy | 28% | — |
| Random | 10% | — |

## Reproducibility notes

- **Stages 2 and 3** are fully reproducible from the released sample data and code.
- **Stage 1** evaluation metrics are reproducible from the pre-generated seed lists stored in the sample data; the forward-chaining inference engine itself runs in the iOS app. The complete rule knowledge base is released in `data/rules/`.
- Results on the 50-trip sample will differ from the paper's 604-trip figures due to sample size. Full-dataset metrics are reported in the paper.
- Random seeds are fixed (`seed=42`) in the sampling script and cross-validation code.
